OPTION B: AtLeastOnce (Recommended - Retry with Disk Buffer)
=========================================================

Logs are written to a disk-based write-ahead log on each node before being sent. If syslog is down, Vector retries. If Vector restarts, it picks up from where it left off.

      url: tls://wss-syslog.example.com:6514
      tls:
        ca:
          key: ca-bundle.crt
          secretName: wss-syslog-tls

    - name: app-alerts-to-syslog
      inputRefs:
        - application
      outputRefs:
        - wss-syslog
      filterRefs:
        - only-your-namespaces

      type: drop
      drop:
        - test:
            - field: .kubernetes.namespace_name
              notMatches: "project-cc|project-cc-staging"

Behavior when syslog server is down:
- Vector writes logs to disk buffer on the node (write-ahead log)
- Retries sending with exponential backoff: 5s, 10s, 20s, 40s, ... up to 5m max
- When syslog comes back, all buffered logs are delivered
- If Vector pod restarts, it reads from disk buffer and continues
- TRADE-OFF: Some logs may be sent twice after a crash (duplicates)

When the disk buffer fills up (this is a CRITICAL question for your team):

  Behavior 1 - BLOCK (default with AtLeastOnce):
  - Vector stops accepting new log events for the syslog output
  - Backpressure propagates upstream, which can slow down ALL sinks
  - This means your Splunk and Dynatrace forwarding could also be affected
  - Pro: zero log loss as long as buffer has space
  - Con: impacts other log destinations during prolonged outage

  Behavior 2 - DROP NEWEST:
  - Vector drops the newest incoming syslog-destined events
  - Other sinks (Splunk, Dynatrace) continue operating normally
  - Pro: isolates the problem to syslog only
  - Con: you lose the most recent logs during extended outage

  NOTE: The CLF API currently maps AtLeastOnce to block behavior. If you
  need drop-newest isolation, we may need to evaluate whether the CLF tuning
  spec exposes that knob, or whether we need a custom Vector config override.

Node resource impact:
- Disk I/O: each log event is written to disk before sending (write-ahead log)
- Memory: bounded by the collector pod memory limit (recommend 1-2 Gi)
- CPU: minimal incremental overhead (Red Hat best practice: do NOT set CPU limits on Vector)
- Disk space: bounded by buffer max_size (we need your input on acceptable size)




THE PROBLEM AS WE UNDERSTAND IT

Your applications currently send syslog messages directly to the xxx syslog server for real-time production support alerts consumed by yyy monitoring. When the syslog server goes down, messages are lost because there is no retry or buffering mechanism in your application code.


WHAT THE PLATFORM CAN OFFER (AT NO ADDITIONAL LICENSING COST)

OpenShift Logging 6.x includes a ClusterLogForwarder (CLF) resource that uses Vector as the log collector. Vector already runs as a DaemonSet on every node, collecting container logs. We can configure it to forward your application logs to your syslog server with built-in delivery guarantees.

Two delivery modes are available:

1. AtMostOnce (default) - Logs are sent once. If the collector restarts or the destination is unavailable, logs in flight may be lost. This is similar to what your apps do today, but centralized.

2. AtLeastOnce (recommended for your use case) - Logs are buffered to disk on each node before being sent. If the collector restarts or the syslog server is temporarily down, buffered logs are retried and re-sent once the server recovers. The trade-off is that some logs may be duplicated after a crash (the same alert may arrive twice), which is generally acceptable for alerting use cases.

How AtLeastOnce works in practice:
- Your app writes a log line to stdout/stderr as it normally does.
- Vector picks it up from the container log file on the node.
- Before sending to syslog, Vector writes the event to a disk-based write-ahead log on the node.
- Vector sends the log to your syslog server over TCP/TLS.
- If the send fails (server down, network issue), Vector retries with exponential backoff, starting at a configurable minimum interval (e.g. 5 seconds) and capping at a configurable maximum (e.g. 5 minutes).
- If the Vector pod restarts, it reads unsent events from the disk buffer and retries them.
- Once the disk buffer reaches its size limit, behavior depends on configuration: it can either block (stop accepting new events until space frees up) or drop the newest events.

Example: If your syslog server goes down for 10 minutes and your app produces 500 log lines during that window, those 500 lines sit in the disk buffer on each node and are delivered once the server comes back.

IMPORTANT CAVEATS:
- The syslog protocol (TCP) does not provide application-level acknowledgment like HTTP or Kafka. Vector relies on TCP connection status to detect failures. This covers server-down and network-drop scenarios well, but cannot detect scenarios where the syslog server accepts the TCP connection but silently drops the message internally.
- Disk buffers are per-node and bounded. They do not provide infinite retention. If the syslog server is down for a very long time and the buffer fills up, the oldest or newest events will be dropped depending on configuration.
- AtLeastOnce adds disk I/O on every node. Under high log volume with a prolonged outage, this can increase disk and memory pressure on the node, which may affect other workloads sharing that node.


ALTERNATIVE: AZURE EVENT HUBS AS A DURABLE BUFFER

If zero message loss is a hard requirement even during extended outages (hours, not minutes), we can place Azure Event Hubs between the CLF and your syslog server. The flow would be:

  App pods -> Vector (CLF) -> Azure Event Hubs -> Consumer app -> Syslog/WSS server

Event Hubs provides durable, partitioned storage with up to 7 days retention (Standard tier). Even if both the collector and the syslog server are down simultaneously, events are safely stored in Event Hubs and replayed when the consumer recovers.

This option has a recurring Azure cost. Rough estimates for Standard tier:
- 1 Throughput Unit (handles up to 1 MB/s ingress, 1000 events/sec): approximately $22-35/month
- Ingress events: approximately $0.028 per million events
- A lightweight consumer (Azure Function on consumption plan) to read from Event Hubs and forward to syslog: approximately $0-5/month
- Total estimate for moderate alerting volume: approximately $35-70/month

This is the most resilient option but adds infrastructure complexity (Event Hubs namespace, consumer application, monitoring of the consumer).


COST SUMMARY

Option A - CLF with AtLeastOnce: $0 additional licensing or Azure cost. The Logging Operator and Vector collector are included with the OCP subscription. Incremental resource usage on existing nodes is the only impact.

Option B - CLF + Azure Event Hubs: Approximately $35-70/month for Standard tier, depending on volume. Plus engineering effort to build and maintain the consumer component.
