
# OpenShift SRE Improvement Ideas & Proposals

**Author:** Madan | **Date:** April 2026 | **Scope:** OCP on Azure — GitOps, Observability, Storage, Mesh, Automation

-----

## Category 1: Observability & Monitoring

### 1.1 SLO-Based Alerting with Pyrra

**Problem:** Current alerting is cause-based (“CPU high on node X”) rather than symptom-based (“API latency exceeds user expectation”). This generates noise and alert fatigue.

**Proposal:** Deploy Pyrra as a Kubernetes operator on each cluster. Define ServiceLevelObjective CRDs for platform services — ArgoCD sync latency, Loki ingestion success rate, OSSM3 gateway p99 latency, etcd leader election frequency. Pyrra auto-generates PrometheusRecordingRules and multi-burn-rate alerts. A web UI shows error budget burn per service, giving the team a single pane to answer “are we burning budget faster than expected?”

**Effort:** Medium | **Impact:** High — reduces false positives, gives leadership clear SLO dashboards

-----

### 1.2 Platform Health Agent (Go + client-go)

**Problem:** Issues like Loki 403s, stale webhooks from removed operators, and CLF misconfigurations are found reactively — often by support teams first.

**Proposal:** Build the `platform-health-agent` you already started designing: a lightweight Go binary per cluster using client-go informers (not shell scripts) to watch resources in real-time. Project areas: Logging (LokiStack distributor health, CLF pipeline status, Vector CPU/memory), Mesh (Istio revision health, webhook validity, gateway certificates), GitOps (ArgoCD app sync status, repo server health, ApplicationSet controller errors), ODF (Ceph health, NooBaa endpoint HPA, BackingStore connectivity), Cluster (operator degraded conditions, node readiness, cert expiry, etcd health). React frontend on hub cluster aggregates all spoke results via REST API.

**Effort:** High | **Impact:** Very High — proactive detection before tickets are filed

-----

### 1.3 Centralized Multi-Cluster PromQL Runner

**Problem:** Running Red Hat diagnostic PromQL queries requires logging into each cluster individually. Grafana has all datasources but still needs manual cluster switching.

**Proposal:** Productionize the multi-cluster PromQL query tool you built — the script that reads ArgoCD cluster secrets from `openshift-gitops` and queries each cluster’s Prometheus endpoint. Package it as a CronJob producing JSON/CSV reports with threshold-based health scoring. Add an HTML report generator for weekly platform health summaries distributed to management. Eventually integrate with the platform health agent.

**Effort:** Low (already prototyped) | **Impact:** Medium — saves hours of manual cluster hopping

-----

### 1.4 Logging Cost Attribution Dashboard

**Problem:** Dual-destination log forwarding (Splunk + Dynatrace + Loki) multiplies log volume costs, and there’s no visibility into which teams/namespaces generate the most volume.

**Proposal:** Build a Grafana dashboard using Vector metrics (`vector_component_sent_bytes_total`, `vector_component_received_events_total`) grouped by namespace and destination. Overlay with Loki stream metrics (`loki_ingester_memory_streams`, `loki_distributor_bytes_received_total`). Per-namespace byte-rate tables let you identify the top 10 log producers and approach those teams about log level tuning or rate limiting. Add PrometheusRules that alert when any single namespace exceeds a configurable bytes/sec threshold.

**Effort:** Low | **Impact:** Medium — direct cost savings and accountability

-----

### 1.5 Debug Log Rate-Limiting as a Platform Service

**Problem:** Application teams leave DEBUG logging enabled in production, overwhelming Loki distributors and causing ingestion rate-limit errors across the cluster.

**Proposal:** Implement CLF filters with per-namespace rate limiting for DEBUG/TRACE level logs. Create a self-service ConfigMap where teams can temporarily enable full debug logging for their namespace (time-boxed, auto-reverts via CronJob). Document an “emergency circuit breaker” filter that can instantly drop all debug logs cluster-wide if ingestion rates exceed safe thresholds.

**Effort:** Medium | **Impact:** High — prevents the “noisy neighbor” log storms you’ve troubleshot multiple times

## Category 2: GitOps & Automation

### 2.1 Namespace Onboarding Platform (ocp-gitops-platform)

**Problem:** Namespace provisioning today goes through Ansible Tower surveys — slow, no audit trail, no PR-based approval, no structural validation.

**Proposal:** Complete the `ocp-gitops-platform` migration: single-repo, Helm chart at `charts/ocp-proj-onboarding`, input files at `onboarding/<cluster>/<lob>.input.yaml`, ArgoCD ApplicationSet reads input files directly as Helm values. Remaining deliverables: `validate-pr.yml` + `validate_input.py` (schema + regex validation), `provision-resource.yml` + `patch_input_from_form.py` (workflow_dispatch with 10 fields), CODEOWNERS per LOB file. Audit trail = Git history. Approvals = PR reviews. Rollback = git revert.

**Effort:** Medium (largely designed, needs build-out) | **Impact:** Very High — eliminates Tower dependency, self-service for LOBs

-----

### 2.2 Operator Vending Machine

**Problem:** Application teams request operator installs via tickets. No standardized assessment, no self-service, no governance.

**Proposal:** Curated CatalogSource pipeline — only pre-approved operators are available. Kustomize base templates + ApplicationSet + AppProjects + Kyverno policies enforce boundaries. Teams self-serve via RHDH Software Template that generates a PR adding their operator to the curated list. Structural controls (AppProject source/destination restrictions, Kyverno policies on allowed namespaces) eliminate need for per-install manual assessment.

**Effort:** Medium | **Impact:** High — reduces ticket load, standardizes operator lifecycle

-----

### 2.3 GitOps Drift Detection & Remediation Reporting

**Problem:** ArgoCD self-heal fixes drift silently. No visibility into what drifted, how often, or whether it indicates a deeper problem (someone manually patching resources).

**Proposal:** Deploy ArgoCD Notifications with webhook triggers on `on-sync-status-unknown` and `on-health-degraded`. Feed events into a lightweight database (or even a ConfigMap-based log). Build a weekly drift report: which apps drifted, which resources, frequency. Repeated drift on the same resource = someone is manually editing and needs education. Can also feed drift events into the platform health agent dashboard.

**Effort:** Low | **Impact:** Medium — governance visibility, identifies manual intervention patterns

-----

### 2.4 ArgoCD ApplicationSet RollingSync

**Problem:** When a platform-wide change (e.g., new NetworkPolicy template) lands in the Helm chart, ArgoCD syncs all namespaces across all clusters simultaneously. A bad template change could impact hundreds of namespaces at once.

**Proposal:** Enable `RollingSync` progressive strategy on ApplicationSets. Configure it to sync dev clusters first, wait for health checks, then proceed to UAT, then prod. Combine with `templatePatch` using Sprig functions to avoid the CI flattening step in the RIR pattern. This gives you canary-style rollouts for platform changes.

**Effort:** Low | **Impact:** High — blast radius reduction for platform changes

-----

## Category 3: Storage & Data

### 3.1 MCG Self-Service Object Storage

**Problem:** Teams needing S3-compatible storage file tickets, ops team manually creates BackingStores, BucketClasses, and OBCs.

**Proposal:** Complete the MCG self-service architecture you designed: per-team BackingStore (lightweight metadata CRDs, zero resource overhead) backed by Azure Blob containers, BucketClass per team, RBAC scoped so teams can only create OBCs in their own namespace. GitOps-managed from hub ArgoCD using Kustomize overlays per team. Vault Secrets Operator for Azure credential rotation. Teams self-serve OBCs by adding a simple YAML to their app GitOps repo.

**Effort:** Medium (design complete, needs lab validation then rollout) | **Impact:** High — eliminates storage provisioning tickets

-----

### 3.2 Alert Storage: Long-Term Retention with Mimir

**Problem:** Prometheus default retention (15 days on OCP) loses historical alert and metric data. Capacity planning, trend analysis, and post-incident review beyond 15 days is impossible.

**Proposal:** Deploy Mimir (or Thanos Receive) with remote-write from each cluster’s Prometheus. Store in Azure Blob Storage for 365-day retention. This enables year-over-year capacity trending, long-term SLO tracking, and historical incident correlation. Cost is primarily Azure Blob storage — cheap at object storage rates.

**Effort:** High | **Impact:** High — unlocks capacity planning, long-term SLO reporting

-----

## Category 4: Service Mesh (OSSM3)

### 4.1 Mesh Namespace Onboarding Automation

**Problem:** Adding a namespace to the service mesh requires manual creation of PeerAuthentication, NetworkPolicy, Sidecar resources — all needing versioned revision labels (e.g., `test-v1-24-3`) that change with Istio upgrades.

**Proposal:** Extend the ArgoCD Matrix generator + `rev-ns-input.yaml` pattern you already deployed. Teams add their namespace to the input file via PR. ApplicationSet auto-generates all mesh resources with correct revision labels. On Istio upgrade, update the revision in one place and ArgoCD propagates everywhere. Add a RHDH Software Template as the frontend.

**Effort:** Low (pattern exists, needs productionization) | **Impact:** Medium — eliminates mesh onboarding toil

-----

### 4.2 Mesh Health Diagnostics

**Problem:** Stale ValidatingWebhookConfigurations from previous operator installations (ROSA/Crown remnants) cause silent failures. Istio revision mismatches break sidecar injection without obvious errors.

**Proposal:** Add mesh-specific checks to the platform health agent: validate all webhooks point to running services, verify IstioCNI node affinity exclusions are correct, check that PeerAuthentication selectors match current revision labels, confirm gateway certificate expiry. Alert on any webhook pointing to a non-existent service (the issue that caused stuck `istio-system` namespace termination).

**Effort:** Low (fits into health agent framework) | **Impact:** Medium — prevents repeat of past mesh incidents

-----

## Category 5: Security & Compliance

### 5.1 Secret Pruner as a Platform Service

**Problem:** `openshift-kube-apiserver` accumulates thousands of rotated secrets (e.g., `localhost-recovery-client-token-*`), degrading etcd performance and increasing backup sizes.

**Proposal:** Productionize the secret pruner (currently at v2.5): inclusion-based (whitelist) filtering, ConfigMap-driven retention policies, concurrency controls, token scoping. Deploy as a weekly CronJob via ArgoCD to all clusters. Review-only mode for sensitive namespaces (`kube-system`, `openshift-etcd`, `openshift-config`). Dashboard showing secret counts over time per namespace to demonstrate ongoing cleanup effectiveness.

**Effort:** Low (built, needs final deployment automation) | **Impact:** Medium — etcd health, backup size reduction

-----

### 5.2 Certificate Expiry Pre-Alerting

**Problem:** Certificate renewals are handled by operators, but failures are silent until services break.

**Proposal:** PromQL-based certificate expiry alerting for all TLS secrets across the cluster. Query: `certmanager_certificate_expiration_timestamp_seconds - time() < 30*24*3600` for cert-manager certs. For platform certs, parse `openshift-kube-apiserver` and `openshift-ingress` secrets. Alert at 30/14/7 day thresholds. Include in platform health dashboard.

**Effort:** Low | **Impact:** Medium — prevents surprise certificate outages

-----

### 5.3 RBAC Audit & Drift Report

**Problem:** Over time, manual `oc adm policy` commands create RBAC bindings outside of GitOps control. No visibility into what’s managed vs. manually created.

**Proposal:** Weekly CronJob that compares actual ClusterRoleBindings and RoleBindings against what ArgoCD manages (label `app.kubernetes.io/managed-by: argocd`). Report shows unmanaged bindings per namespace. Over time, migrate all RBAC to GitOps and use this report to track progress toward 100% GitOps-managed RBAC.

**Effort:** Low | **Impact:** Medium — compliance visibility, governance improvement

-----

## Category 6: Developer Experience

### 6.1 RHDH as the Self-Service Portal

**Problem:** Platform services (namespace creation, operator requests, mesh onboarding, storage provisioning) are accessed via different ticket queues, wikis, and manual processes.

**Proposal:** RHDH becomes the single entry point. Software Templates for: namespace onboarding (triggers `provision-resource.yml`), mesh namespace enrollment (updates `rev-ns-input.yaml`), MCG storage request (adds team to `gitops-mcg-storage` overlays), operator install request (adds to curated catalog). Each template generates a PR, goes through approval, and ArgoCD syncs. The RHDH catalog shows all provisioned resources with live health status via the Kubernetes plugin.

**Effort:** High (RHDH already deployed, templates need building) | **Impact:** Very High — unified developer experience, toil reduction

-----

### 6.2 Dockerfile-to-Image Build Pipeline via RHDH

**Problem:** Teams building custom container images have no standardized build pipeline. Images are built locally or via ad-hoc Jenkins jobs.

**Proposal:** RHDH template that takes a GitHub Enterprise repo with a Dockerfile, creates an OpenShift Pipeline (Tekton), builds the image, pushes to Artifactory, and optionally makes it available as an AAP Execution Environment. PR-based approval before production push. The template and pipeline are reusable across teams.

**Effort:** Medium | **Impact:** Medium — standardizes image builds, improves supply chain security

-----

## Category 7: Azure Infrastructure

### 7.1 Azure Storage Policy Automation

**Problem:** Azure storage policies (lifecycle management, access restrictions) are applied manually per storage account.

**Proposal:** Azure Automation Runbook (or better, Azure Policy with DeployIfNotExists) that discovers storage accounts by tag or naming pattern and applies standard policies. Run on schedule via Managed Identity — no manual login needed. Tag-based targeting (`RequiresPolicy: true`) means new storage accounts automatically get policies.

**Effort:** Low | **Impact:** Medium — compliance automation

-----

### 7.2 Node Auto-Scaling Tuning

**Problem:** Cluster Autoscaler defaults may not be optimal for your workload patterns. Scale-up is too slow, scale-down too aggressive, or node churn wastes money.

**Proposal:** Review and tune Cluster Autoscaler parameters per cluster: `scaleDownUtilizationThreshold` (default 0.5 — may be too aggressive), `scaleDownDelay` (increase to prevent churn), `maxNodeProvisionTime` (account for Azure VM spin-up time). Create per-environment MachineSet profiles: smaller instances for dev (cost savings), larger for prod (performance). Document recommended HPA/VPA configurations for common workload patterns as part of namespace onboarding guide.

**Effort:** Low | **Impact:** Medium — cost optimization + performance

## Category 8: FinOps & Cost Optimization

### 8.1 OpenCost / Kubecost for Namespace-Level Cost Attribution

**Problem:** No visibility into which teams/namespaces consume the most compute, storage, and network resources. Azure billing shows VM-level costs but not Kubernetes-level attribution. Finance asks “what does Team X cost us?” and there’s no answer.

**Proposal:** Deploy OpenCost (CNCF project, free) or Kubecost on each cluster. OpenCost allocates real Azure VM costs down to namespace, pod, and label level using actual Azure pricing APIs. Expose per-LOB cost dashboards via Grafana. Integrate with namespace onboarding so every new namespace automatically gets cost labels. Weekly cost reports per LOB sent to stakeholders. This shifts cost accountability to application teams — the core FinOps “inform” phase.

**Effort:** Medium | **Impact:** High — direct visibility into cloud spend, enables chargeback/showback

-----

### 8.2 Shift-Left Cost Estimation in Namespace Onboarding

**Problem:** Teams request namespaces with ResourceQuotas but have no idea what those quotas will cost on Azure.

**Proposal:** Add a cost estimation step to the `provision-resource.yml` workflow. When a team requests a namespace with specific CPU/memory quotas, a Python script calculates the estimated monthly Azure cost based on current VM instance pricing and adds it as a comment on the PR. Approvers see “This namespace at full quota would cost ~$X/month.” Prevents over-provisioning at the request stage rather than discovering waste after the fact.

**Effort:** Low | **Impact:** Medium — cost awareness before provisioning

-----

### 8.3 Dev/UAT Cluster Scale-Down Scheduling

**Problem:** Dev and UAT clusters run 24/7 at full capacity but are only actively used during business hours (EST). Nights and weekends waste ~65% of compute spend.

**Proposal:** CronJob-based MachineSet scaling: scale dev/UAT worker MachineSets to minimum (e.g., 2 nodes) at 8 PM EST, scale back up at 7 AM EST. Use `timeZone: America/New_York` in CronJob spec. Protect prod clusters — they never scale down. For workloads that must persist overnight, label them and use node affinity to keep them on the minimum nodes. Estimated savings: 30-50% on dev/UAT Azure compute.

**Effort:** Low | **Impact:** High — immediate Azure cost reduction

-----

## Category 9: Chaos Engineering & Resilience

### 9.1 Platform Chaos GameDays with Kraken

**Problem:** DR procedures and runbooks exist on paper but are never tested. When a real incident happens (like the etcd crash you recovered from), the team discovers gaps in the runbook live.

**Proposal:** Quarterly Chaos GameDays using Kraken (Red Hat’s chaos testing tool for OpenShift/Kubernetes). Start with controlled experiments on lab/dev clusters: kill an etcd member, drain a master node, simulate Azure zone failure, delete a CLF pipeline, kill the ArgoCD repo server. Document what breaks, how long recovery takes, and update runbooks. Progressively increase scope to UAT. Never run unannounced chaos on prod — the goal is learning, not surprise.

**Effort:** Medium | **Impact:** High — validates DR readiness, builds team confidence

-----

### 9.2 Automated etcd Backup Verification

**Problem:** etcd backups run on schedule but nobody verifies they’re restorable until a disaster happens. A corrupt backup discovered during recovery is catastrophic.

**Proposal:** Weekly CronJob that takes the latest etcd snapshot, restores it into a temporary etcd container (not connected to the cluster), and validates it can start and serve reads. Report pass/fail to the platform health dashboard. Alert if backup age exceeds 24 hours or if the verification job fails. This turns “we have backups” into “we have verified backups.”

**Effort:** Medium | **Impact:** Very High — prevents the worst-case disaster scenario

-----

### 9.3 Azure Availability Zone Failover Testing

**Problem:** Canada Central and Canada East clusters are DR pairs, but actual failover has never been tested end-to-end. Unclear if DNS failover, storage replication, and ArgoCD re-targeting work as designed.

**Proposal:** Scheduled DR drill: simulate loss of Canada Central by cordoning all nodes in one zone, verify workloads shift to Canada East via DNS/traffic management, validate data consistency, measure RTO/RPO, then restore. Document gaps. Run quarterly. Eventually automate the drill as a Tekton pipeline.

**Effort:** High | **Impact:** Very High — proves DR actually works

-----

## Category 10: Supply Chain Security

### 10.1 Container Image Signing with Cosign/Sigstore

**Problem:** No cryptographic verification that container images running in your clusters were built by your CI pipelines. A compromised Artifactory or man-in-the-middle attack could inject malicious images.

**Proposal:** Integrate Cosign signing into Tekton/OpenShift Pipelines via Tekton Chains. Every image built by your pipelines gets signed with a keypair stored in `openshift-pipelines` namespace. Deploy a Kyverno ClusterPolicy (or use OCP 4.16+ ClusterImagePolicy) that rejects unsigned images in prod namespaces. Start with “audit” mode (log violations, don’t block), then move to “enforce” after teams adopt the pipeline.

**Effort:** Medium | **Impact:** High — closes a critical supply chain gap

-----

### 10.2 SBOM Generation and Vulnerability Tracking

**Problem:** When a critical CVE drops (like Log4Shell), there’s no way to quickly answer “which of our running images are affected?”

**Proposal:** Generate SBOMs (Software Bill of Materials) using Syft during image builds. Attach SBOMs to images in OCI registries via Cosign. Run Grype vulnerability scans against SBOMs in CI. Store SBOM artifacts in Artifactory. When a new CVE is announced, run a single query against stored SBOMs to identify all affected images and namespaces. Integrates with Red Hat ACS (Advanced Cluster Security) if available.

**Effort:** Medium | **Impact:** High — instant CVE impact assessment, regulatory compliance readiness

-----

### 10.3 Image Provenance with SLSA Framework

**Problem:** Even with signed images, there’s no proof that an image was built from a specific Git commit by a specific pipeline (provenance).

**Proposal:** Add SLSA Level 2 provenance generation to build pipelines. Use `slsa-github-generator` or Tekton Chains’ built-in SLSA attestation. Attach provenance attestations to images. Enforce in admission controllers: “only deploy images with provenance from org/repo on main branch.” This is increasingly becoming a regulatory requirement (US Executive Order 14028, EU Cyber Resilience Act).

**Effort:** Medium | **Impact:** Medium — compliance, audit readiness

-----

## Category 11: OpenTelemetry & Unified Observability

### 11.1 Jaeger to OpenTelemetry + Tempo Migration

**Problem:** Red Hat deprecated the Jaeger operator (removed from catalog November 2025). If you’re still using Jaeger for distributed tracing, you’re on an unsupported component.

**Proposal:** Migrate to Red Hat build of OpenTelemetry Operator + TempoStack. Deploy OTEL Collector as the unified telemetry receiver (traces via OTLP, metrics via Prometheus scraping, logs via FileLog receiver). TempoStack backed by Azure Blob Storage for trace storage. GitOps-managed via ArgoCD. This also positions you for the next idea.

**Effort:** Medium | **Impact:** High — removes deprecated component, future-proofs tracing

-----

### 11.2 OTEL Collector as Unified Telemetry Gateway

**Problem:** Telemetry is fragmented: Vector for logs, Prometheus for metrics, potentially Jaeger/Tempo for traces. Three collection pipelines, three sets of configuration, three sets of failure modes. Forwarding to external destinations (Splunk, Dynatrace) requires configuring each pipeline separately.

**Proposal:** Deploy OTEL Collector in gateway mode (centralized StatefulSet) as a single forwarding layer to external destinations. Keep Vector for in-cluster log collection (it’s optimized for that), keep Prometheus for in-cluster metrics scraping, but route both through OTEL Collector for external export. Benefits: single configuration point for external destinations, unified filtering/sampling before export (reduce Splunk/Dynatrace ingest costs), single place to add new destinations.

**Effort:** High | **Impact:** High — cost reduction on external observability, architectural simplification

-----

### 11.3 Auto-Instrumentation for Application Teams

**Problem:** Application teams don’t instrument their code with tracing. When a cross-service issue occurs, there’s no distributed trace to follow.

**Proposal:** Use OTEL Operator’s auto-instrumentation injection. Teams add a single annotation (`instrumentation.opentelemetry.io/inject-java: "true"`) to their Deployment and the operator injects the OTEL agent sidecar. No code changes required for Java, Python, Node.js, .NET. Traces flow to TempoStack automatically. Offer this as a self-service option via RHDH Software Template. Start with a pilot team, demonstrate value with a real cross-service trace, then promote adoption.

**Effort:** Low per team | **Impact:** Medium — unlocks distributed tracing without dev effort

-----

## Category 12: Policy & Governance

### 12.1 CEL ValidatingAdmissionPolicy for Resource Guardrails

**Problem:** Teams can create resources that violate platform standards (e.g., containers without resource limits, pods with `latest` tag, services with overly permissive NetworkPolicies). Currently no enforcement beyond documentation.

**Proposal:** Use Kubernetes-native CEL ValidatingAdmissionPolicies (GA in K8s 1.30+, OCP 4.16+). No external policy engine needed. Start with: require resource requests/limits on all containers, block `latest` image tags in non-dev namespaces, require specific labels on all Deployments, enforce MachineSet naming conventions. Audit mode first, then enforce after a grace period. GitOps-managed policies via ArgoCD.

**Effort:** Low | **Impact:** Medium — platform standards enforcement without Kyverno/OPA overhead

-----

### 12.2 Quota Right-Sizing Recommendations

**Problem:** Teams request generous ResourceQuotas during onboarding but actual usage is often 20-30% of quota. This wastes cluster capacity and blocks other teams.

**Proposal:** Weekly CronJob that compares actual namespace CPU/memory usage (from Prometheus) against quota allocations. Generate a “right-sizing report” showing namespaces where usage is <30% of quota for 30+ consecutive days. Send to LOB owners with recommendations to reduce. Can integrate with the FinOps cost attribution dashboard — show “you’re paying for 8 CPU but using 1.5.”

**Effort:** Low | **Impact:** Medium — capacity reclamation, cost reduction

-----

### 12.3 Network Policy Audit & Visualization

**Problem:** NetworkPolicies exist per namespace but nobody has a holistic view of inter-namespace communication paths. When a new microservice can’t talk to another, debugging is guesswork.

**Proposal:** Deploy a NetworkPolicy visualization tool (e.g., Cilium’s Hubble if using OVN-Kubernetes with eBPF, or build a custom tool using `oc get networkpolicy -A` + graph generation). Render a cluster-wide network topology showing allowed/denied flows. Integrate with platform health dashboard. When a team onboards to mesh, this shows them their current connectivity posture.

**Effort:** Medium | **Impact:** Medium — reduces networking debugging time, security visibility

-----

## Category 13: Toil Reduction & Process

### 13.1 Runbook Automation with RHDH + AAP Integration

**Problem:** Incident runbooks are static wiki pages. During an incident, an engineer reads the runbook and manually executes commands. This is slow and error-prone under pressure.

**Proposal:** Convert top-10 most-used runbooks into Ansible Automation Platform job templates. Trigger them from RHDH via a Software Template or custom action — one-click “restart LokiStack distributor,” “drain and uncordon node,” “scale MachineSet.” Guard with approval workflows for destructive actions. RHDH becomes the operational console, AAP provides the execution engine, Git provides the audit trail.

**Effort:** Medium | **Impact:** High — incident MTTR reduction, standardized response

-----

### 13.2 Cluster Upgrade Pre-Flight Checks

**Problem:** OCP upgrades occasionally break things — deprecated APIs, incompatible operators, insufficient node capacity. Pre-upgrade checks are manual and inconsistent.

**Proposal:** Automated pre-flight check pipeline (Tekton or GHA Job) that runs before every cluster upgrade: scan for deprecated APIs using `oc adm upgrade` recommendations, check all operator compatibility with target version, verify etcd health, check node capacity for rolling upgrade (needs N+1 capacity), verify backup freshness, check certificate expiry dates. Fail the upgrade pipeline if any check fails. Generate a “readiness report” for change management approval.

**Effort:** Medium | **Impact:** High — de-risks upgrades, builds confidence

-----

### 13.3 Postmortem & Incident Tracking in Git

**Problem:** Incidents are discussed in Slack/Teams, sometimes a ticket is updated, but structured postmortems are rare. Lessons learned are lost.

**Proposal:** Create a `incidents/` directory in a Git repository. After every P1/P2, open a PR with a structured postmortem template (timeline, root cause, impact, action items, 5-whys). PR review ensures quality. Action items become GitHub Issues linked to the postmortem. Monthly review of open action items. Git history becomes the institutional memory of incidents. Low-tech, high-value.

**Effort:** Very Low | **Impact:** Medium — cultural improvement, prevents repeat incidents

-----

### 13.4 Platform Engineering Newsletter / Changelog

**Problem:** Platform changes (new Helm chart templates, updated policies, new self-service features) are communicated ad-hoc. Teams don’t know what’s available.

**Proposal:** Monthly “Platform Changelog” — a markdown document in the GitOps repo (or a RHDH TechDocs page) that lists: new features, breaking changes, deprecations, upcoming changes. Auto-generated from Git commit messages tagged with `[changelog]`. Distributed via email or Teams. Establishes the platform team as a product team with a release cadence.

**Effort:** Very Low | **Impact:** Low-Medium — developer experience, adoption of platform features
